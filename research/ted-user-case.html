01 — User Profile

Who is the user?

The user is a business development professional or domain expert working in a specialist industry — in this case, digital health and cardiac telemonitoring. They have deep knowledge of their market: the clinical terminology, the key players, the product categories, the regulatory context.

They are not a data engineer or procurement officer. They do not know SPARQL, RDF, or the eProcurement Ontology. They have never heard of CPV codes. Their goal is business intelligence — understanding market activity, identifying opportunities, tracking competitor wins — not technical data extraction.

They work in English and operate across multiple European markets, including countries where they do not speak the local language (e.g., Italy, Germany, France).

02 — Context & Goal

What does the user want to do?

The user wants to answer a straightforward business question:

“Show me all EU public tenders related to heart failure telemonitoring in Italy over the past 5 years — who bought, how much, which suppliers won.”

Typical user query

This is a market sizing and opportunity identification task. The user expects to type a topic in plain English, select a country and date range, and receive structured, readable results — similar to how they would use Google, LinkedIn Sales Navigator, or a market intelligence platform.

The underlying data exists in TED (Tenders Electronic Daily), the EU’s official procurement database. Over 750,000 notices are published annually across 27 member states.

03 — Friction Points

Where does the current system fail the user?

CRITICAL — 01

The user knows their domain, not the classification system

TED organises all data around CPV codes — a bureaucratic EU classification taxonomy created for procurement officers. The user knows “heart failure telemonitoring”; they have no way to know this maps to CPV 33123200. There is no intuitive bridge between industry language and the required vocabulary.

CRITICAL — 02

Cross-language blindness

The user operates in English. Italian tenders are written in Italian, German tenders in German. Keyword search is therefore useless — the words the user would naturally type do not appear in the documents they are looking for. There is no built-in translation layer in the system.

CRITICAL — 03

No feedback loop — silent failure

When a query returns zero or few results, the user cannot tell if this means “nothing exists” or “you asked the wrong question.” There is no relevance score, no “did you mean,” no confidence indicator. The system provides silence with no guidance on how to improve the query.

HIGH — 04

CPV codes are too coarse for niche themes

Even if the user finds a relevant CPV code, the classification is broad. “Cardiac monitoring devices” covers everything from a basic ECG to a cloud telemonitoring platform. Conversely, a telemonitoring tender may be filed under “health IT services” or “miscellaneous health services.” The theme is scattered across unrelated codes with no consistent pattern.

HIGH — 05

Query requires technical expertise the user should not need

To extract data properly, the user must learn SPARQL, understand RDF graph structure, know URI formats for country codes, navigate the eProcurement Ontology namespace, and implement pagination. This is infrastructure work, not the user’s job. The technical barrier excludes the exact professionals who need this data most.

MEDIUM — 06

Results arrive in raw procurement language

Even when results are retrieved successfully, notice titles and descriptions are in the originating country’s language, use procurement jargon, and reference internal contract codes. The user cannot quickly assess relevance without manual translation and context-building for each result.

04 — Current User Journey

What does the user actually go through today?

1

User identifies a business need

Wants to know the size of the HF telemonitoring procurement market in Italy. Searches for a reliable data source and lands on TED / OP-TED GitHub.

2

Tries keyword search — fails immediately

Types “telemonitoring” or “heart failure monitoring” into TED search. Gets few or zero results from Italy because Italian tenders use Italian keywords. Has no indication of why results are sparse.

3

Discovers CPV codes are required — learning curve begins

Learns that data is structured around CPV codes, not keywords. Must now research which CPV codes are relevant to their niche. Uses external tools (SIMAP, LLM assistants) to build a shortlist. Time cost: 1–3 hours.

4

Discovers SPARQL is required to query properly

The TED UI search is insufficient for structured multi-field queries. Learns about the SPARQL endpoint. Must learn RDF graph structure, eProcurement Ontology predicates, URI formats for countries, and pagination. Time cost: several hours to days for a non-technical user.

5

Runs queries — uncertain about completeness

Gets some results but cannot verify if the dataset is complete. Is unsure whether low result counts reflect a small market, wrong CPV codes, or incorrect query syntax. No feedback mechanism exists to validate coverage.

6

Manually translates and filters results

Exports data in Italian. Uses Google Translate or DeepL to read notice titles and descriptions. Manually judges relevance for each row. Significant time investment for what should be an automated filtering task.

05 — Core Insight

The fundamental problem

Product Insight

TED was built for compliance, not intelligence. It enables procurement officers to publish notices and auditors to verify them. It was never designed for a business development professional asking “What is the activity level of the HF telemonitoring market in Italy over 5 years?” That question is entirely reasonable. The system makes it unnecessarily hard to answer.

The gap between what the user knows (domain expertise, industry language, business questions) and what the system requires (CPV codes, SPARQL, multi-language awareness) is the core product opportunity. A new system should eliminate this gap entirely — the user should never have to learn procurement infrastructure to get procurement intelligence.

07 — Definition of Success

When has the system solved the problem?

The system succeeds when a domain expert with no knowledge of CPV codes, SPARQL, or Italian can:

1

Type a theme in plain English

“Heart failure telemonitoring” — no taxonomy knowledge required.

2

Select country and date range from a UI

Italy, 2021–2026. No URI formats or RDF syntax required.

3

Receive results in English within seconds

Structured rows with buyer, value, date, translated title. Immediately readable and actionable.

4

Export to CSV and use in their workflow

No manual translation, no data cleaning. Ready to import into a spreadsheet or dashboard.

“Total time from question to usable data: under 2 minutes. Today, it takes 2–3 days for a non-technical domain expert.”
